# دليل تطوير وتوثيق تطبيق "كروت الدحشة" (Dahsha Cards) 🚀

تطبيق **كروت الدحشة** هو نظام أندرويد متكامل ومصمم خصيصاً لإدارة وتوزيع كروت الشحن الإلكترونية تلقائياً بمجرد قراءة رسائل الإشعارات (SMS) الواردة من المحافظ المالية والبنوك الإلكترونية اليمنية مثل (محفظة جيب، محفظة جوالي، كريمي، ون كاش، إم فلوس، حاسب) وتوجيه الكروت المناسبة للعملاء فورياً.

---

## 1. بنية وهيكلية التطبيق (App Architecture) 🏗️

تم بناء التطبيق باتباع أفضل ممارسات تطوير تطبيقات أندرويد الحديثة:
* **لغة البرمجة**: Kotlin بالكامل.
* **واجهة المستخدم**: Jetpack Compose المبنية على نظام **Material Design 3 (M3)** مع دعم الوضع المظلم التلقائي واللمسات الجمالية المتكاملة.
* **إدارة الحالة والمنطق (State & Logic)**: نمط **MVVM (Model-View-ViewModel)** باستخدام `ViewModel` و `StateFlow` لتدفق البيانات بشكل حي ومتزامن مع واجهات العرض.
* **تخزين البيانات وتدفقها**: قاعدة بيانات **Room Database** المحلية مع استخدام `Flow` لضمان تحديث واجهات التطبيق تلقائياً فور حدوث أي عملية إيداع أو شحن جديدة.
* **استقبال الإشعارات التلقائي**: `BroadcastReceiver` يستمع لرسائل الـ SMS الواردة بالخلفية ويقوم بتحليلها فوراً، حتى وإن كان التطبيق مغلقاً.
* **نظام التنبيهات الداخلي (In-App Notification Bus)**: آلية مخصصة عبر `SharedFlow` تسمى `NotificationBus` لتنبيه المستخدم فوراً بداخل التطبيق وتفعيل نافذة منبثقة تفاعلية عند استخراج الكود بنجاح.

## 2. هيكلة وتفاصيل قاعدة البيانات (Database Schema Definition) 🗄️

يستخدم التطبيق قاعدة بيانات **Room SQLite** محلية ومصممة بعناية تحت اسم `dahsha_database` وتتكون من 5 جداول رئيسية لتغطية الدورة البرمجية بالكامل:

### أ. جدول الكروت المتوفرة (`cards`)
يحتوي على مخزون كروت الشحن المرفوعة في التطبيق والموزعة حسب الفئات:
* **`id`** (`Int` - Primary Key AutoGenerate): المعرف الفريد للكارت.
* **`category`** (`Int`): فئة الكارت (مثال: 100، 200، 250، 300، 500 ريال يمني).
* **`code`** (`String`): كود كارت الشحن الفعلي الذي سيتم إرساله للعميل.
* **`username`** (`String`): اسم المستخدم الاختياري لشبكات المايكروتك.
* **`password`** (`String`): رمز المرور الاختياري لشبكات المايكروتك.
* **`used`** (`Boolean`): حالة الكارت (تكون `false` كارت جديد غير مستخدم، وتتحول إلى `true` عند شحنه وإرساله للعميل بنجاح).

### ب. جدول الإيداعات المستلمة (`deposits`)
يسجل كل عملية إيداع مالي تم استلامها عبر الرسائل لضمان التتبع المالي الكامل:
* **`id`** (`Int` - Primary Key AutoGenerate): الرقم التسلسلي للإيداع.
* **`phone`** (`String`): رقم هاتف العميل أو الكود التعريفي الخاص به الذي أرسل الدفعة.
* **`amount`** (`Int`): المبلغ المودع المستقطع.
* **`walletType`** (`String`): مصدر المحفظة التي أرسلت المبلغ (مثل: "جيب"، "جوالي"، "كريمي"، إلخ).
* **`isShared`** (`Boolean`): حالة تسليم الكارت (تكون `true` إذا تم تسليمه بنجاح تلقائياً، وتكون `false` إذا كانت الدفعة معلقة وتنتظر تفويضاً يدوياً).
* **`cardDetails`** (`String`): تفاصيل كود الكارت الذي تم توزيعه لصاحب هذا الإيداع.
* **`createdAt`** (`Long`): الطابع الزمني وتاريخ استلام الإيداع بدقة بالملي ثانية.

### ج. جدول العمليات الناجحة (`transactions`)
أرشيف العمليات المكتملة وتوثيق تاريخ الشحن لكل عميل:
* **`id`** (`Int` - Primary Key AutoGenerate): المعرّف الفريد للعملية.
* **`phone`** (`String`): رقم هاتف العميل المستلم.
* **`amount`** (`Int`): فئة الشحن المستلمة.
* **`cardCode`** (`String`): كود الشحن الفعلي الموزع.
* **`walletType`** (`String`): اسم المحفظة المستخدمة لإرسال القيمة.
* **`createdAt`** (`Long`): تاريخ ووقت تأكيد العملية وتوزيع الكارت.

### د. جدول التفويضات المعلقة (`pending_approvals`)
مخصص للعمليات التي تحتاج تدخل المستخدم وتأكيده اليدوي (مثلاً: عندما تكون الخدمة التلقائية متوقفة، أو عند نفاد فئة معينة من الكروت، أو عند ورود دفعة برمز حساب بدلاً من رقم هاتف مباشر):
* **`id`** (`Int` - Primary Key AutoGenerate): المعرّف التسلسلي للطلب المعلق.
* **`phone`** (`String`): رقم الهاتف أو كود الحساب الوارد في الرسالة.
* **`amount`** (`Int`): قيمة العملية المالية المستلمة.
* **`walletType`** (`String`): اسم المحفظة المصدر.
* **`createdAt`** (`Long`): تاريخ ووقت تعليق العملية.
* **`isAccountCode`** (`Boolean`): علم يشير إلى ما إذا كان العميل قد دفع عبر كود حساب مخصص (مثل: "12345") بدلاً من رقم هاتف مباشر، مما يستلزم مطابقة حسابه لإرسال الكود للرقم الأصلي.
* **`depositId`** (`Int`): لربط الطلب المعلق مع إدخال الإيداع الأصلي بجدول `deposits`.

### هـ. جدول مطابقة حسابات العملاء الاستثنائيين (`customer_mappings`)
يتيح لأصحاب المحلات ربط أكواد الحسابات الفريدة بأرقام هواتف عملائهم الفعلية لتسهيل الإرسال التلقائي:
* **`id`** (`Int` - Primary Key AutoGenerate): الرقم التعريفي للتسجيل.
* **`customerUniqueId`** (`String`): الرمز الفريد للعميل المستخرج من رسالة الإيداع (مثل الاسم أو معرف الحساب).
* **`basicPhone`** (`String`): رقم هاتف العميل الأساسي المعتمد لإرسال رسائل الكروت إليه.
* **`customerName`** (`String`): الاسم الحقيقي للعميل لتمييزه بواجهة التطبيق.
* **`walletType`** (`String`): المحفظة المعتمدة للعميل (القيمة الافتراضية "جيب").

## 3. منطق فك تشفير وتحليل رسائل الـ SMS تلقائياً (Unique SMS Parsing Logic) 🧪

يمتلك التطبيق معالجاً متطوراً في `SmsParser` يستخدم تعابير برمجية منتظمة (Regex) مخصصة لكل محفظة في اليمن:

### الصيغ المعتمدة للمحافظ:

1. **محفظة جيب (Jeeb)**:
   * **النمط الافتراضي**: `500ر.ي تم استلامها من المحفظة - 777777777`
   * **التعبير البرمجي (Regex)**: `(\d+)ر\.ي.*-(.+)`

2. **محفظة جوالي (Jawali) - التحديث الأحدث**:
   * **النمط المدعوم والمعتمد**: `استلمت مبلغ 5100 YER من 700070034 رصيدك هو 5300`
   * **التعبير البرمجي (Regex)**: `استلمت مبلغ (\d+) YER من\s*(.+?)\s*(?:رصيدك|\(|\s*$|\n)`
   *(يستخلص هذا التعبير المبلغ "5100" ورقم الهاتف "700070034" ويتفادى النص الإضافي مثل رصيدك الحالي).*

3. **محفظة كريمي (Kuraimi)**:
   * **النمط**: `أودع/احمد جابر حسن المنتصر لحسابك مبلغ 7400 رصيدك 7450 YER`
   * **التعبير البرمجي**: `أودع/([^/\n]+)\s+لحسابك\s+مبلغ\s+(\d+)`

4. **محفظة ون كاش (One Cash)**:
   * **النمط**: `استلمت 200.00 من بدر حسين حنين رصيدك1,252.59 ر.ي`
   * **التعبير البرمجي**: `استلمت\s+(\d+(?:\.\d+)?)\s+من\s+(.+?)\s+رصيدك`

5. **محفظة إم فلوس (M-Floos)**:
   * **النمط**: `تم إيداع مبلغ 3000 YER من: قابوس سعيد المرجع: 42556517`
   * **التعبير البرمجي**: `تم إيداع مبلغ\s+(\d+)\s+YER\s+من:\s*(.+?)\s*(?:المرجع|\s*$|\n)`

6. **محفظة حاسب (Haseb)**:
   * **النمط**: `اضيف 200ر.ي تحويل مشترك رص:41692.31ر.ي من 120025`
   * **التعبير البرمجي**: `اضيف\s+(\d+)\s*ر\.ي\s+تحويل مشترك.*من\s+(\d+)`

7. **صيغ الرسائل المخصصة (Custom Templates)**:
   * تتيح للمستخدم كتابة النمط الخاص به بمرونة متناهية في الإعدادات، مثل كتابة الصيغة واستبدال القيم بمتغيرات (`%amount`, `%phone`, `%account`) ويقوم محرك التطبيق تلقائياً بتحويلها خلف الكواليس إلى Regex ديناميكي فائق الدقة ومطابقتها.

## 4. دورة حياة المعالجة اليدوية والتلقائية للرسائل (SMS Lifecycle & Flow) 🔄

```
 [ رسالة SMS واردة من محفظة ماليّة ]
               │
               ▼
   [ SmsReceiver يستقبل الإشعار ]
               │
        (هل الخدمة مفعلة؟)
         ├── لا ──> تجاهل الرسالة وعدم معالجتها
         └── نعم
               │
               ▼
     [ SmsParser.parse() ] ───(فشل التحليل)───> تجاهل
               │
         (نجاح التحليل) ───> استخراج (المبلغ والمُعرّف والمحفظة)
               │
               ▼
  [ البحث عن كارت غير مستخدم فئة المبلغ بجدول cards ]
               │
     ┌─────────┴─────────┐
(تم العثور على كارت)  (لم يتم العثور على كارت)
     │                   │
     ▼                   ▼
(هل العميل برقم متاح؟)  [ إضافة الطلب كمعلق في pending_approvals ]
 ├── نعم (رقم مباشر) ────> إرسال كود الكارت فورياً عبر SMS للعميل (إذا التوزيع التلقائي مفعل)
 │                       وإدراج العملية في deposits كعملية تمت مشاركتها بنجاح (isShared = true).
 │                       وإطلاق حدث حي إلى NotificationBus.
 │
 └── لا (رمز حساب) ─────> يتم مراجعة جدول customer_mappings:
                         * إذا وجد رقم هاتف مربوط بالحساب: إرسال الكارت تلقائياً للرقم المربوط.
                         * إذا لم يجد رقم مربوط: وضع الطلب في pending_approvals وتصنيفه كحساب استثنائي بانتظار التفويض.
```

## 5. ميزة الإشعارات والمنبثقات التفاعلية داخل التطبيق (In-App Live Notifications) 🔔

تم توفير تجربة مستخدم تفاعلية فائقة السلاسة عبر إضافة **حافلة الأحداث المشتركة (NotificationBus)**:
1. عند استلام رسالة شحن جديدة وتوليد الكود بنجاح (سواء تلقائياً بالكامل أو بعد موافقة يدوية)، يطلق التطبيق حدثاً حياً يحمل البيانات (`NewCardExtractedEvent`).
2. تلتقط واجهة التطبيق الرئيسية `MainDashboardScreen` هذا الحدث تلقائياً وتعرض نافذة تنبيه تفاعلية مميزة (`AlertDialog`) في غضون أجزاء من الثانية.
3. **ميزات النافذة المنبثقة**:
   * تظهر تهنئة حية وجميلة: `🎉 تم استخراج كرت جديد بنجاح`.
   * تظهر جميع تفاصيل العملية (المبلغ المستلم، رقم العميل، المحفظة، حالة إرسال الرسالة التلقائي).
   * تعرض كود الكرت المستخرج بخط عريض وبلون لافت مريح للعين.
   * زر تفاعلي ذكي: **"مشاركة وفتح تطبيق المحفظة 🚀"**، يقوم بنسخ الكود تلقائياً للحافظة وفتح تطبيق المحفظة المعني لتسهيل التواصل الفوري.
   * زر **"نسخ الكود فقط 📋"** و **"إغلاق ✖"**.

## 6. معلومات تقنية ومعرّف التطبيق الفريد (Technical Specs) 🛠️

* **اسم حزمة التطبيق (Application ID)**: `com.aistudio.dahshacards.uylxtb`
* **مستوى الـ SDK الأدنى**: API 24 (أندرويد 7.0 فما فوق)
* **مستوى الـ SDK المستهدف**: API 36
* **الصلاحيات الأساسية المطلوبة بالمنفذ (AndroidManifest)**:
  * `android.permission.RECEIVE_SMS` (لاستقبال وقراءة الرسائل)
  * `android.permission.SEND_SMS` (لإعادة توجيه الكروت للعملاء)
  * `android.permission.READ_SMS` (للوصول للرسائل النصية وتدقيقها)
* **تقنيات البناء المستعملة**:
  * **Room** لإدارة البيانات المحلية.
  * **KSP (Kotlin Symbol Processing)** لتوليد أكواد الـ DAO بكفاءة عالية وسرعة بناء قياسية.
  * **Jetpack Compose Navigation** للتنقل الآمن والسهل بين التبويبات والشاشات.
